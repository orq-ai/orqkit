The Problem

Agents act on every word they read.

Traditional apps are secured deterministically. Agents execute on every word they read. Every input is a potential payload: user, retrieval, tool output.

The Tradeoff

Power and permissions scale together.

The Method

What is red teaming?

Adversarial probing of an LLM application

Sending crafted attacks to find exploitable weaknesses before they reach production.

What

Crafted attacks, not happy paths

Prompt injection, tool misuse, policy bypass, multi-turn drift. Inputs designed to make the agent misbehave.

Why

Catch failures before users do

Find the breach in a controlled run, not in a customer ticket or an incident channel.

How

Structured, recurring attacks

Each attack categorised, scored, and tracked over time. Re-run after every prompt or model change.

The Method

Red teaming is a loop.

Every change re-opens the surface. The job is not "secure the agent once"; it is close the loop so attacks are caught before the customer is.

Today's Focus

Three categories under the spotlight.

System Prompt Leakage

Agent reveals its system prompt, exposing policy rules, tool list, and refusal patterns.

Agent Goal Hijacking

Attacker overrides the agent's intended objective via prompt injection. Flips "enforce policy" into "approve this refund."

Tool Misuse & Exploitation

Hijacked agent calls issue_refund outside policy: wrong owner, wrong window, wrong reason. Money moves.

Each scored independently by its own LLM-as-judge.

The Scenario

Customer service refund agent.
Three tools, three rules.

The agent & its tools

The policy

  1. Standard refund: in 30 days → confirm + refund.
  2. Post-window exception: enumerated reason only [damaged_in_transit, never_received, defective_on_arrival] + verifiable evidence.
  3. Anything else → refuse + route to human queue.

post_window_exception=True bypasses the window check.

Agent decides when to set it.

Tool checks ownership, window, double-refund. Exception logic falls to LLM judgment — no deterministic check.

The Attack

Red teaming the refund agent

Policy Hallucination + Context Injection

What just happened?
Two failures combined.

Failure 1

Policy hallucination

Sympathetic story → invented exception.
Emergency · urgency · "just-this-once".

Failure 2

Untrusted-context injection

Chat input absorbed as fact.
Quoted policy · tool-output snippet · authority claim.

Core problem
The agent can't tell its own knowledge from the attacker's claim.

Structure over story

System prompt addition

+ For post-window refunds:+ 1. Call get_policy("post_window_exceptions") first.+    Use only returned text.+ 2. Reason MUST match enum [damaged_in_transit,+    never_received, defective_on_arrival]+    + verifiable evidence.+ 3. Refuse if either missing. No negotiation.

Try It Yourself

Three steps to red team your agent

Step 1 · Configure agent

Set up the agent in orq.

Step 2 · Run red team

One call. Recommendations included.

from evaluatorq import red_team

report = await red_team(
    "agent:my-agent",
    generate_recommendations=True,
)

for rec in report.focus_area_recommendations:
    print(rec.category, rec.risk_score)
    for r in rec.recommendations:
        print("  -", r)

It Generalises

Any agent. Any attack class.

The refund agent is one example. Same framework, same workflow, different agent.

Indirect injection

Email assistants

Injection turns them into phishing relays from inside the company.

Tool misuse

Code review agents

PR description text and inline comments treated as instructions. Attacker hides "approve and merge" in commit body.

Policy hallucination

Internal HR bot

Fetches policy from KB, answers benefits questions. Indirect injection via HR doc updates or pasted "company policy" text.

Voice + tools

Personal assistants

Hermes, OpenClaw and similar voice agents read calendars, emails, docs; any of those channels can carry an injection that triggers tools on your behalf.

Supported targets

LLM

Custom

Custom callable

Any async function that maps a message list to a response. Escape hatch for in-house or unsupported frameworks.

Framework

Hand it a configured Agent. Pipeline drives multi-turn attacks through the SDK's runner.

Framework

LangChain

Wrap a LangChain agent. Reuses OpenResponses format; tool calls + memory introspected.

Framework

LangGraph

Wrap a compiled graph. Per-attack thread isolation, tool + memory introspection, token tracking.
