Types UML — evaluatorq, redteam, agent simulation

Four regions share one bridge: AgentResponse. Evaluatorq core owns the unified output contract; redteam owns the AgentTarget protocol, orchestration models (Turn-canonical OrchestratorResult), and job I/O boundary (JobOutputPayload); backends/integrations provide target implementations with history; agent simulation owns persona/scenario/result models. OrqResponsesTarget is dual-role with a partial-stateless contract (sim __call__ stateless, redteam send_prompt stateful). RES-844 proposal section at the bottom captures the planned move of AgentTarget into evaluatorq.contracts and the optional stateless respond(messages) path.

Key invariants

  • OrchestratorResult.turns is the canonical record — conversation, final_response, n_turns are derived. Do not read from conversation if you need per-turn attacker/target pairing.
  • AgentResponse.output preserves order. .tool_calls is a filtered view. Use .output when text/reasoning ordering matters.
  • ToolCallOutputItem has two IDs: id (item identity) and call_id (tool call correlation across the function_call + function_call_output pair).
  • AgentTarget.new() must return a fresh instance with clean state and a new memory_entity_id for persistent-memory targets.
  • AttackInfo.delivery_methods is always a list[DeliveryMethod], never singular.
  • JobOutputPayload → RedTeamResult is the required conversion path. Do not construct RedTeamResult directly from OrchestratorResult.
  • JobOutputPayload.extra='allow' — forward-compatible with new job output fields.
  • Two distinct TokenUsage types — redteam has calls + arithmetic, sim is slim. Not interchangeable.
  • OrqResponsesTarget.__call__ resets _previous_response_id per turn (full history passed by sim). Mixing __call__ and send_prompt on the same instance corrupts threading.
  • RedTeamResult.execution is None for static pipeline runs.
  • passed=True means RESISTANT (attack failed to elicit the target behavior). Counterintuitive — read carefully.

Gotchas

  • OrchestratorResult.conversation is derived. It unpacks turns into a flat list[Message]. Modifying it has no effect on turns.
  • Two Message types exist. redteam.contracts.Message (5 fields, supports tool role) vs simulation.types.Message (2 fields, no tool support). Do not swap them.
  • OpenAIModelTarget vs OpenAIAgentTarget. OpenAIModelTarget calls raw Chat Completions — stateless model, no tools/handoffs. OpenAIAgentTarget calls OpenAI Agents SDK — stateful via _history, supports tool use and multi-agent handoffs.
  • AgentResponse.text returns the last text item. Multi-text outputs lose earlier chunks. Read .output for full content.
  • ToolCallOutputItem.arguments is a JSON string. Use .arguments_dict — bad JSON silently becomes empty dict.
  • TokenUsage.total_tokens=0 fallback. from_completion computes prompt+completion when provider reports zero total, preventing silent under-counting.
  • LLMConfig.retry_config returns {} in OpenAI mode. The retry param is ORQ-specific and rejected by OpenAI.
  • Legacy migration is implicit. scope → vulnerability_domain, framework='mixed' → 'hybrid', SendResult → AgentResponse. All silent at validator time.

FAQ

Why does OrchestratorResult use turns: list[Turn] instead of conversation: list[Message]?

The canonical record preserves attacker/target pairing per turn — Turn holds both the AttackerResponse (with usage + truncation flag) and the AgentResponse (with full output items). A flat Message list loses this structure. conversation is a convenience property for code that only needs the OpenAI-format history.

Why two TokenUsage types?

Sim's predates the redteam variant; redteam needed calls, from_completion, and arithmetic operators for aggregation across multi-continuation ORQ calls. Merging requires migrating all sim aggregation sites.

Why does JobOutputPayload exist between OrchestratorResult and RedTeamResult?

It's the job I/O wire format — stored, returned via API, deserialized. extra='allow' makes it forward-compatible. RedTeamResult is the report-layer view built after eval enrichment and agent metadata are merged in.

What's the difference between OpenAIModelTarget and OpenAIAgentTarget?

OpenAIModelTarget: calls raw Chat Completions API, takes a model name string, accumulates _history as plain message pairs. OpenAIAgentTarget: calls OpenAI Agents SDK Runner.run(), takes an Agent object, threads history via to_input_list(), supports tools and multi-agent handoffs. Use the model target for bare model red-teaming; use the agent target for Agents SDK apps.

Which AgentTarget implementation should I use?

ORQ-hosted agents → ORQAgentTarget. Raw OpenAI model → OpenAIModelTarget. LangGraph graph → LangGraphTarget. OpenAI Agents SDK → OpenAIAgentTarget. Vercel AI SDK HTTP endpoint → VercelAISdkTarget. Anything else → CallableTarget (wraps any fn(str) → str).

Why is OrqResponsesTarget dual-role?

Single implementation serves sim (__call__ returning str) and redteam (send_prompt returning AgentResponse). Avoids duplicating Responses v3 client logic, retry, and usage accounting. State semantics differ: __call__ is stateless w.r.t. self — never reads or writes _previous_response_id / _accumulated_usage, the sim runner owns the transcript and tracks usage independently. send_prompt is stateful — threads via _previous_response_id and accumulates usage on the instance. Internally both paths flow through a shared pure _call_responses_api helper via two private wrappers (_invoke_stateless, _invoke_stateful). A future ADR may split into two classes.

Proposal — RES-844 protocol unification

RES-844 promotes the AgentTarget protocol family out of redteam/backends/base.py so both red-team and simulation can consume the same contract, and unifies the sim/redteam target shapes. Below is the scope we landed on after design review (including an adversarial pass from architecture-reviewer) and three Orq-docs/codebase audits.

Current state on main

  • AgentTarget + Supports* family + BackendBundle + helpers live in packages/evaluatorq-py/src/evaluatorq/redteam/backends/base.py (lines 1–289). Nothing else in the repo defines them.
  • OrqResponsesTarget already satisfies the protocol structurally — it carries memory_entity_id, send_prompt, new(), and tests/redteam/test_orq_responses_target_as_agent_target.py (17/17 passing) asserts is_agent_target(OrqResponsesTarget) is True.
  • simulation/runner/simulation.py:73 defines its own TargetAgent protocol with a different method shape (respond(messages) -> str) — no shared contract with redteam today.
  • ORQAgentTarget already uses the Responses route via agents.responses.create → /v3/router/responses (NOT the legacy /v3/agents/execute-task endpoint). State threading is via Orq-specific task_id + memory_entity_id; tool-call continuation is a client-side loop that re-POSTs synthetic {kind:'tool_result'} parts.

  • redteam/backends/base.py becomes a thin re-export shim. All red-team-side imports (backends/orq.py, backends/openai.py, backends/registry.py, runtime/orq_agent_job.py) keep working unchanged.
  • Add an optional stateless method to AgentTarget: async respond(messages: list[ChatMessage]) -> AgentResponse. Implement on OrqResponsesTarget by folding the current __call__ path into it; __call__ stays as a thin back-compat shim returning .text. Red-team backends do not implement respond in PR A.
  • Add adapter as_target_callback(target: AgentTarget) -> Callable[[list[ChatMessage]], Awaitable[str]] in contracts.py. Internally extracts the last user message and calls send_prompt when respond is absent. Enables passing any red-team backend through the sim's target_callback entrypoint — closes acceptance criterion #2.
  • Replace simulation/runner/simulation.py:73's ad-hoc TargetAgent protocol with from evaluatorq.contracts import AgentTarget. One protocol, one source.
  • Round-trip smoke test in tests/common/test_target_protocol.py: red-team backend through sim, sim target through red-team.
  • Do not touch ChatMessage. Stays as chat-completions {role, content: str} in simulation/types.py:217. Tool-call / tool-result transcript fidelity is PR B scope.

PR B — separate L ticket (deferred)

True architectural unification on the Orq Responses input model. Should be filed as a new ticket so RES-844 stays small and reviewable.

  • Adopt openai.types.responses.ResponseInputItemParam (or a sibling discriminated InputItem union in contracts.py following the proven OutputMessage pattern at contracts.py:80–83). Delete the hand-rolled TypedDicts in openresponses/types.py.
  • Migrate all red-team backends to a single async respond(transcript: list[InputItem]) -> AgentResponse. Deprecate send_prompt with one minor's grace period.
  • Move ORQAgentTarget's tool-continuation loop OUT of the target and into the orchestrator/caller, so targets stay pure functions of (transcript) -> AgentResponse. Server returns aggregated usage per call; local accumulation today is an artifact of the client-side loop, not a requirement.
  • Introduce a ThreadHandle (or equivalent) only if red-team needs fork/rollback for crescendo attacks. Current behavior (per-attack new() instance) is enough for most cases.
  • Rewrite call sites in redteam/runner.py:722, 1436, redteam/adaptive/pipeline.py:379, redteam/adaptive/orchestrator.py:652, and every integration (langgraph, callable, vercel_ai_sdk, openai_agents).

Why split A vs B

  • PR A is purely additive + a directory move. No call-site rewrites, no protocol semantics change. Reviewable in one sitting.
  • PR B is the real architectural move — it touches every backend and every call site, and it requires extending the transcript data model. Smuggling it into a "promote protocol family" ticket would balloon RES-844 to L and block the downstream sim tickets.
  • The adversarial review surfaced four load-bearing concerns (silent messages ignore, concurrency footgun on OrqResponsesTarget, tool-result representation gap, common/-vs-contracts placement). PR A only addresses placement; the other three are scoped to PR B explicitly so they get the attention they need.

Decisions log

  • Bandwidth is not a concern. Even if a stateless caller-owned-history flow re-uploads the full transcript every turn, we don't care. Rules out the cost-based objection to PR B's direction.
  • previous_response_id is not a token-cost optimization. LLM input tokens are billed identically whether you thread via previous_response_id or pass the full transcript inline. Only client→server upload bytes differ.
  • ORQAgentTarget already uses the Responses route. The legacy /v3/agents/execute-task endpoint is not in play. agents.responses.create SDK method maps to /v3/router/responses.
  • Server-side usage is aggregated per call. Local _accumulated_usage on OrqResponsesTarget and ORQAgentTarget exists only because of the client-side tool-continuation loop. Move the loop to the caller (PR B) and local accumulation drops out.
  • ChatMessage cannot represent Responses input items. It is OpenAI chat-completions shape ({role, content: str}) with no tool-call / tool-result / multi-part content. PR B copies the OutputMessage discriminated-union pattern from contracts.py:80–83 rather than extending ChatMessage in-place.
  • Protocol home is evaluatorq.contracts, not a new common/ module. AgentResponse already lives there; making a second "shared stuff" directory creates a fragmentation we'll regret.
